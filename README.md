# InvoiceFlow

**SMU IS213 · Enterprise Solution Development · AY2025/26**

InvoiceFlow is an invoice factoring marketplace where businesses list invoices for auction and investors bid to finance them. The platform handles the full lifecycle: invoice listing, competitive bidding with anti-snipe protection, fund disbursement via Stripe, loan tracking with Temporal durable workflows, and debt resolution with choreography-driven event fan-out.

## Environment Setup

### 1. Copy the example env file

```bash
cp .env.example .env
```

### 2. Fill in required secrets

Open `.env` and replace these four placeholder values — everything else defaults correctly for Docker Compose:

#### `JWT_SECRET`

```bash
python -c "import secrets; print(secrets.token_hex(32))"
```

Paste the output into `.env`.

#### `STRIPE_SECRET_KEY` and `STRIPE_WEBHOOK_SECRET`

InvoiceFlow uses Stripe's **test (sandbox) mode** — no real money moves.

**Secret key (`sk_test_...`):**
1. Log in to [dashboard.stripe.com](https://dashboard.stripe.com)
2. Click **Developers** (top-right corner)
3. Click **API keys** in the left sidebar
4. Copy the **Secret key** — it starts with `sk_test_`

**Webhook signing secret (`whsec_...`):**

The webhook secret is generated by the Stripe CLI when you start a local listener. Install the CLI first:

```bash
# macOS
brew install stripe/stripe-cli/stripe

# Windows (Chocolatey)
choco install stripe-cli

# Or download the binary: https://docs.stripe.com/stripe-cli
```

Then open a **new terminal window from the root of this repo** (`invoiceflow/`) and run:

```bash
stripe login                  # authenticate once — opens a browser to confirm
stripe listen --print-secret  # prints your whsec_... secret and exits
```

Copy the printed `whsec_...` value into `.env`. When you later want to receive live webhook events during development, open a **new terminal from the repo root** (keep it running alongside `docker compose`) and run:

```bash
stripe listen --forward-to localhost:5013/api/webhooks/stripe
```

#### `RESEND_API_KEY`

InvoiceFlow uses [Resend](https://resend.com) to send transactional emails (outbid alerts, auction results, etc.).

**Getting a free API key:**
1. Sign up at [resend.com](https://resend.com) (free, no credit card)
2. In the dashboard left sidebar, click **API Keys**
3. Click **Create API Key**, set permissions to **Full access**, and click **Add**
4. Copy the key immediately — it is only shown once

**Free tier limitation — important:** Resend's free plan only allows sending emails to the **one email address you registered with**. You cannot send to other addresses unless you verify a custom domain. This means:
- Notifications work end-to-end only when the recipient happens to be your own Resend account email
- If you just want to confirm the notification pipeline is wired up, trigger an action whose notification goes to your own email and check your inbox
- To lift this restriction, verify a domain in the Resend dashboard (Domains → Add Domain)

All other variables (service URLs, DB hostnames, RabbitMQ, MinIO, Temporal addresses) are pre-set to Docker Compose service names and work out of the box.

## How to Build & Run

### Full stack (recommended)

```bash
docker compose up --build
```

This builds all service images and starts:
- All 13 microservices + 2 wrappers + Temporal worker
- MySQL databases (one per service, auto-initialised from `databases/*/init.sql`)
- RabbitMQ, MinIO, Temporal server, KONG API Gateway

Wait for all containers to be healthy:

```bash
docker compose ps
```

### Rebuild a single service

Use this when iterating on one service without restarting everything:

```bash
docker compose up --build <service-name>

# Examples:
docker compose up --build user-service
docker compose up --build payment-service
docker compose up --build temporal-worker
```

### Run in detached mode (background)

By default, `docker compose up` occupies your terminal and streams logs from every container. Adding `-d` (detach) starts all containers in the background so your terminal is free:

```bash
docker compose up --build -d
```

The stack is running the same way — you just don't see the logs inline. To check on things afterwards:

```bash
docker compose ps                        # see which containers are running and their health status
docker compose logs -f <service-name>    # stream logs for one specific service
docker compose logs -f                   # stream logs for all services
docker compose down                      # stop all containers and remove them
```

### Frontend (development mode, outside Docker)

```bash
cd frontend
npm install
npm run dev        # starts Vite dev server at http://localhost:5173
```

## Database Migrations & Seed Data

Databases are initialised automatically when their containers first start. Each service has an init script at `databases/<service>-db/init.sql` that creates tables and seeds any required reference data.

**No manual migration step is required** for a fresh `docker compose up --build`.

If you need to re-run init scripts (e.g. after a schema change), destroy and recreate the volumes:

```bash
docker compose down -v          # removes containers AND named volumes
docker compose up --build       # re-creates databases from init.sql
```

> **Warning:** `down -v` deletes all data in the databases. Only use this to reset a dev environment.

## Architecture Overview

```
Frontend (React + Vite)
      │
      ▼
Nginx :80/:443           ← SSL termination, WebSocket proxy, static routing
      │
      ▼
KONG API Gateway :8000   ← JWT validation, rate limiting, CORS, API routing
      │
      ├─→ User Orchestrator :5015      → ACRA Wrapper (seller UEN validation)
      │                                 → User Service (account creation)
      │
      ├─→ Invoice Orchestrator :5010   → Invoice Service, Marketplace Service, ACRA Wrapper
      │                                 → Temporal (start AuctionCloseWorkflow)
      │
      ├─→ Bidding Orchestrator :5011   → Bidding Service, Marketplace Service
      │                                 → Payment Service (gRPC: LockEscrow)
      │                                 → Stripe Wrapper (wallet top-up)
      │                                 → Temporal (signal extend_deadline / start WalletTopUpWorkflow)
      │
      ├─→ Loan Orchestrator :5012      → Payment Service (gRPC: GetLoan, UpdateLoanStatus)
      │                                 → Stripe Wrapper (loan repayment)
      │                                 → RabbitMQ (publish loan.repaid)
      │
      └─→ User Service / Notification Service (direct, read-only)

Temporal Worker  ← polls invoiceflow-queue
  ├── AuctionCloseWorkflow  (timer → anti-snipe → 10-step settlement → start LoanMaturityWorkflow)
  ├── LoanMaturityWorkflow  (sleep until due → check repaid → mark OVERDUE if not)
  └── WalletTopUpWorkflow   (credit wallet → publish wallet.credited)

RabbitMQ  exchange: invoiceflow_events  (topic)
  bid.placed, bid.outbid
  auction.closed.winner, auction.closed.loser, auction.closing.warning, auction.extended
  loan.repaid   → 4 consumers (Invoice, Payment, User, Notification)
  loan.overdue  → 4 consumers (Invoice, Payment, User, Notification)
  wallet.credited

Monitoring Stack
  Prometheus :9090   ← scrapes /metrics from all services + RabbitMQ
  Grafana    :3001   ← dashboards, alerting (datasources: Prometheus, Loki, Tempo)
  Loki       :3100   ← log aggregation (fed by Promtail)
  Promtail              ← ships container logs to Loki
  Tempo      :3200   ← distributed tracing (OTLP gRPC :4317)
```
